## Day 1 - Lambda Setup and Local Validation

### Objective

Create the producer Lambda code locally and validate that the function handles both valid and invalid event payloads before deploying to AWS.

### Lambda Function Purpose

The producer Lambda receives an event payload, validates required fields, adds a timestamp, and returns a structured response.

Required fields:

- `userId`
- `eventType`
- `source`

### Successful Local Test

Command:

```bash
python3 -c 'import importlib.util; spec = importlib.util.spec_from_file_location("app", "lambda/producer/app.py"); app = importlib.util.module_from_spec(spec); spec.loader.exec_module(app); print(app.lambda_handler({"body":"{\"userId\":\"aj-001\",\"eventType\":\"login\",\"source\":\"local-test\"}"}, None))'
```

Result:

The Lambda returned `statusCode: 200`.

### Local Validation Failure Test

Command:

```bash
python3 -c 'import importlib.util; spec = importlib.util.spec_from_file_location("app", "lambda/producer/app.py"); app = importlib.util.module_from_spec(spec); spec.loader.exec_module(app); print(app.lambda_handler({"body":"{\"userId\":\"aj-001\",\"source\":\"local-test\"}"}, None))'
```

Result:

The Lambda returned `statusCode: 400` because `eventType` was missing.

### Lesson

Testing locally before deploying helps catch syntax errors and logic issues early. This also gives a clean baseline before troubleshooting AWS service integrations.

## Day 3 - Lambda to DynamoDB Storage Test

### Objective

Persist valid API events into DynamoDB.

### Final Day 3 Flow

```text
API Gateway POST /events
→ Producer Lambda
→ DynamoDB EventTable
→ CloudWatch Logs
```

### DynamoDB Table Design

Table: `EventTable`

Partition key:

```text
userId
```

Sort key:

```text
timestamp
```

This design allows multiple events to be stored for the same user while preserving event order by timestamp.

### Successful DynamoDB Write

Command:

```bash
curl -X POST "https://YOUR_API_ID.execute-api.us-east-2.amazonaws.com/events" \
  -H "Content-Type: application/json" \
  -d '{"userId":"aj-001","eventType":"login","source":"curl-day-3"}'
```

Result:

The API returned:

```text
Event accepted and saved to DynamoDB
```

### DynamoDB Verification

Opened DynamoDB, selected `EventTable`, explored table items, and confirmed an item was stored with:

- `userId`: `aj-001`
- `eventType`: `login`
- `source`: `curl-day-3`
- `timestamp`: generated by Lambda

### IAM Permission Required

The producer Lambda execution role required:

```text
dynamodb:PutItem
```

Scoped to:

```text
arn:aws:dynamodb:us-east-2:YOUR_ACCOUNT_ID:table/EventTable
```

### Lesson

Lambda can persist validated API events into DynamoDB when the execution role has scoped `dynamodb:PutItem` permission. This demonstrates least-privilege access to a specific table.

### Kinesis Region Correction

Issue:

The first Kinesis stream was accidentally created in `us-east-1`, while the rest of the lab resources were in `us-east-2`.

Root Cause:

The AWS Console region selector was set to N. Virginia instead of Ohio.

Fix:

Deleted the `us-east-1` stream and recreated `event-stream` in `us-east-2`.

Correct stream ARN:

```text
arn:aws:kinesis:us-east-2:YOUR_ACCOUNT_ID:stream/event-stream
```

Lesson:

For clean troubleshooting, keep API Gateway, Lambda, DynamoDB, Kinesis, Secrets Manager, and CloudWatch in the same AWS Region.

### Lambda Code Paste Error

Issue:

API Gateway returned:

```text
Internal Server Error
```

CloudWatch showed:

```text
NameError: name 'EOF' is not defined
```

Root Cause:

Shell heredoc text, specifically `EOF`, was accidentally pasted into the AWS Lambda console editor.

Fix:

Removed non-Python shell text from `lambda_function.py`, redeployed the Lambda, and retested the API.

Lesson:

AWS Lambda code editor should contain only Python code. Shell commands like `cat > file << 'EOF'` belong only in the local terminal.

---

### Consumer Lambda IAM Issue

Issue:

When adding the Kinesis trigger to `event-consumer-console`, AWS returned an error saying the Lambda role could not access `event-stream`.

Error summary:

```text
Cannot access stream arn:aws:kinesis:us-east-2:YOUR_ACCOUNT_ID:stream/event-stream.
```

Root Cause:

The consumer Lambda execution role did not have Kinesis read permissions.

Fix:

Added an inline IAM policy allowing:

- `kinesis:GetRecords`
- `kinesis:GetShardIterator`
- `kinesis:DescribeStream`
- `kinesis:DescribeStreamSummary`
- `kinesis:ListShards`
- `kinesis:ListStreams`

The stream-specific actions were scoped to:

```text
arn:aws:kinesis:us-east-2:YOUR_ACCOUNT_ID:stream/event-stream
```

Lesson:

The producer Lambda needs `kinesis:PutRecord` to write to the stream. The consumer Lambda needs read permissions to poll records from the stream.

### Successful Consumer Test

Command:

```bash
curl -X POST "https://YOUR_API_ID.execute-api.us-east-2.amazonaws.com/events" \
  -H "Content-Type: application/json" \
  -d '{"userId":"aj-001","eventType":"stream-test","source":"curl-day-4-consumer"}'
```

Result:

The consumer Lambda was triggered by Kinesis and processed the stream record.

CloudWatch evidence:

```text

```

### Lesson

Kinesis decouples event intake from event processing. The producer accepts and publishes events, while the consumer processes stream records asynchronously.

### Secrets Manager IAM Failure

Issue:

The producer Lambda initially failed when calling `secretsmanager:GetSecretValue`.

Error:

```text
AccessDeniedException: not authorized to perform secretsmanager:GetSecretValue
```

Root Cause:

The producer Lambda execution role did not have permission to read the secret.

Fix:

Added a scoped inline IAM policy allowing:

```text
secretsmanager:GetSecretValue
```

for:

```text
arn:aws:secretsmanager:us-east-2:YOUR_ACCOUNT_ID:secret:event-pipeline/app-config-*
```

Lesson:

Lambda functions can only access AWS services if their execution role allows the required action on the correct resource. Secrets Manager access should be scoped to the specific secret instead of all secrets.

---

### Secret Key Mismatch

Issue:

After IAM was fixed, the Lambda returned:

```text
appMode=unknown
```

Root Cause:

The secret did not contain the expected key name `appMode`.

Fix:

Updated the secret key/value to:

```text
appMode=training
```

Lesson:

IAM permissions can be correct while the application still returns wrong values if the secret content or key names do not match the code.

---

### Safe Logging Practice

The Lambda confirmed the secret was loaded by printing only metadata:

```text
Loaded app config from Secrets Manager. Keys available: ['appMode']
```

The Lambda did not print secret values directly.

### Lesson

Secrets Manager should be accessed through IAM-scoped permissions. Lambda logs should confirm safe operational behavior without exposing sensitive values.

|---|---|

| Consumer Lambda | Processes Kinesis records and logs event details |
| CloudWatch Logs | Provides operational evidence and troubleshooting details |
| IAM | Controls least-privilege access between services |

## Interview Explanation

I built a serverless event pipeline using API Gateway, Lambda, DynamoDB, Kinesis, CloudWatch, IAM, and Secrets Manager. The API accepts event payloads through `POST /events`, validates them in Lambda, enriches them with app config from Secrets Manager, saves them to DynamoDB, and publishes them to Kinesis. A consumer Lambda processes the stream asynchronously and logs the processed events to CloudWatch.

During the build, I troubleshot real issues including a Kinesis Region mismatch, an accidental `EOF` paste error in Lambda code, missing Kinesis read permissions for the consumer Lambda, a missing Secrets Manager permission for the producer Lambda, and a secret key mismatch. Each issue was identified through CloudWatch logs or AWS service errors, fixed, retested, and documented.

This gave me practical experience with event-driven architecture, least-privilege IAM, serverless troubleshooting, safe logging, and AWS operational discipline.
